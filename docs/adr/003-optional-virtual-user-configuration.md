# ADR-003: 仮想ユーザー設定のオプショナル化

## ステータス
承諾済み

## 日付
2026-07-19

## コンテキスト
bswitchは従来、`[default]`セクションで`writer_user`/`writer_api_key_ref`と`reader_user`/`reader_api_key_ref`の両方を必須としていた。
しかし、ユーザーによってはwrite/admin権限のみ使用し、read用の仮想ユーザーを用意しないケースがある（逆も同様）。
両方を必須にすると、片方しか使わないユーザーにも不要な仮想ユーザーの作成とAPIキーの管理を強いる��とになる。

### 検討した選択肢
1. **暗黙フォールバック（未設定時はwriterをreaderとしても使う）**
   - 概要: reader未設定時にwriter側の値を自動で流用する
   - メリット: config記述が最小で済む
   - デメリット: ユーザーが意図せずread権限でwriter用ユーザーが使われる。設定ミスに気づきにくい

2. **同一値の明示設定を公式サポート**
   - 概要: 両方に同じ値を書けばOKとドキュメントする。`get_status`表示バグだけ修正
   - メリット: コード変更が最小
   - デメリット: 4項目に同じ値を2回書く冗長さが残る。意図が設定から読み取れない

3. **writer/readerそれぞれ独立にオプショナル化（遅延バリデーション）**
   - 概要: config読み込み時はペア整合性と最低1ペアの存在のみ検証��、実際に必要になった時点（switch実行時）でエラーにする
   - メリット: 未使用側の設定が不要。エラーメッセージが具体的（「read権限を使おうとしたがreader_userが未設定」）
   - デメリット: 影響箇所が多い（models/config/keys/switcher全体）。型が`str | None`になり扱いが複雑化

## 決定
私たちは **writer/readerそれぞれ独立にオプショナル化（遅延バリデーション）** を使用します。

### 決定理由
- 暗黙フォールバックは「設定ミスを隠す」方向に働く。安全 > シンプルさの設計指針に反する
- 同一値の明示設定は技術的に動作するが、`get_status`の表示問題やAPI呼び出しの無駄が残り、ユーザーに設定の意図を説明しにくい
- 遅延バリデーション方式はコード変更量は大きいが、エラーメッセージが明確で、設定ファイルからユーザーの意図が読み取れる構成になる

## 結果

### ポジティブ
- writerのみ/readerのみの構成で、不要な仮想ユーザーの作成・管理が不要になる
- 設定ミス（read権限を使おうとしたがreader未設定）が具体的なエラーメッセージで即座に判明する
- 既存の両方設定済み構成は一切影響を受けない��後方互換性あり）

### ネガティブ
- `DefaultConfig`のフィールドが`str | None`になり、利用箇所で型ガードが必要（`type: ignore[assignment]`を2箇所使用）
- `_exclusive_remove`でNone側のスキップ判定が入り、���ードが若干複雑化

### トレードオフ
- config読み込み時ではなくswitch実行時にエラーを出す（遅延バリデーション）ため、設定ファイルを書いた時点では問題に気づかない。ただしプロファイルのpermissionと照合して初めて意味をなす検証のため、config単体で検出するのは過剰

## コンプライアンス
- `writer_user`と`writer_api_key_ref`は必ず��アで設定する（片方だけはConfigError）
- `reader_user`と`reader_api_key_ref`は必ずペアで設定する（片方だけはConfigError）
- writer/readerの両方が未設定の場合はConfigError（最低1ペアは必須）
- `permission = read`のプロファイルをswitchで選択した場合、reader_userが未設定ならSwitcherError
- `permission = write/admin`のプロファイルをswitchで選択した場合、writer_userが未設定ならSwitcherError
