# ADR-001: レートリミットに配慮したstate.json差分ベースの権限切り替え方式

## ステータス
承諾済み

## 日付
2026-07-18

## コンテキスト

bswitchは、仮想ユーザー（writer/reader）をBacklogプロジェクトへ「必要な時だけ・排他的に」参加させる権限スイッチャーである。排他性を保証するため、switch時には以前参加していたプロジェクトからの除名（スイープ）が必要になる。

当初実装は「config定義済みの**全**プロジェクトから両仮想ユーザーを除名する」方式だった。この方式はstate.jsonに依存しない自己修復性を持つ一方、実運用のプロファイル数（nプロジェクト）では nプロジェクト × 2ユーザー × 2操作（ユーザー除名・管理者除名）= **n*4回の更新系APIコール** をswitchのたびに消費する。

また、当初実装はstate.jsonの保存をコマンド完了時の1回のみ行っていたため、API操作成功後・保存前の失敗（1PasswordのAPIキー解決失敗、後続API呼び出しのレートリミット、Ctrl+C等）で「実際は参加済みなのに記録がない」状態が発生し得た。除名対象をstate.json由来に絞る場合、この記録漏れは「仮想ユーザーが権限を持ち続けるのに誰も回収しない」事故に直結するため、保存戦略の見直しも同時に必要になった。

### 検討した選択肢

1. **全プロジェクトスイープ（当初実装）**
   - 概要: switchのたびにconfig定義済み全プロジェクトから両仮想ユーザーを除名する
   - メリット: state.jsonの内容に依存しない自己修復性（記録が壊れても毎回全件掃除される）
   - デメリット: 更新系APIをn*4回/switch消費しレートリミットが即枯渇する。プロファイル追加に比例して悪化する

2. **write-ahead記録 + grants由来スイープ**
   - 概要: API操作の前に「付与予定のgrants」をstate.jsonへ保存し、除名対象を前回grants記載のプロジェクトのみに絞る
   - メリット: API成功〜ファイル書き込み間のクラッシュ窓が存在しない（記録が常に実状態の上位集合）
   - デメリット: 途中失敗時に「付与予定だが実際は未付与」の偽記録が残り、次回スイープで無駄な除名APIコールが発生する。記録が実状態を正確に反映しない

3. **こまめな保存 + grants由来スイープ + 継続スキップ（採用）**
   - 概要: 除名・参加のAPI操作が**成功するたびに即座に**state.jsonへ保存する。除名対象は前回grants記載のプロジェクトのみ。さらに「同一プロジェクト・同一仮想ユーザーで継続」と判定できる場合は除名→再参加のサイクル自体を省略する
   - メリット: 記録が常に実状態を反映し偽記録が残らない。同一プロファイルの再switchがAPIコール0回になる。ファイル書き込みはレートリミットを消費しない
   - デメリット: API成功〜ファイル書き込み間（ミリ秒単位）のプロセス強制終了で記録漏れの余地が残る。state.json消失時はswitch単体では自己修復しない

## 決定

私たちは **案3（こまめな保存 + grants由来スイープ + 継続スキップ）** を使用します。

### 決定理由

- 最も希少なリソースはBacklogの更新系レートリミット枠であり、ローカルファイル書き込みは何度実行してもこの枠を消費しない。「APIコール最小化 > 保存回数の節約」という優先順位が明確
- write-ahead方式（案2）と比較して、失敗時に偽記録由来の無駄な除名APIコールが発生しない。記録が常に「実際に成功した操作」だけを反映するため挙動の推論が容易
- 残余リスク（ミリ秒窓のクラッシュ・state.json消失）に対しては、`bswitch release --all` が**全プロジェクトスイープ**として残っており、明示的な回復経路が存在する（当初は `release` の既定動作だったが、日常のレートリミット消費を抑えるため既定をstate.json基準に変更し、全スイープは `--all` に移した）

### 付随する決定（エラーハンドリング方針）

- **delete系API（除名スイープ）**: code 6（NoResourceError: 対象不在）に加えて code 5（UnauthorizedOperationError: 権限なし）も無視して継続する。スイープが権限エラーで止まるとswitch自体が一切できなくなるため
- **add系API（選択したプロジェクトへの参加）**: code 5 でも例外を送出する。ユーザーが明示的に選択したプロジェクトへの参加失敗を隠すと「成功したように見えるが使えないAPIキーがexportされる」偽の成功になるため
- **admin権限の記録順序**: 管理者付与APIの実行**前**に permission="admin" として記録する。付与成功後・記録前に落ちても除名側がadminとして掃除するため安全側。付与に失敗したら直後に "write" へ訂正して保存する

## 結果

### ポジティブ

- switchあたりの更新系APIコール: 旧n*4回 → 通常4〜5回（前回1プロジェクト分の除名+新規参加）
- 同一プロファイルの再switch（新しいターミナルでの環境変数設定・期限更新）: **0回**
- 同一プロジェクト内の権限切り替え: admin→write は1回、write→admin は2回のみ
- 途中失敗・中断しても、参加済みプロジェクトは必ず記録に残り、次回switchの除名スイープで自動回収される
- release / enforce_expired にも同じこまめな保存を適用し、全コマンドで記録と実状態のズレが最小化される

### ネガティブ

- state.jsonが排他制御の唯一の信頼点になる（消失・破損時、switchは参加済みプロジェクトを検知できない）
- API成功直後のプロセス強制終了（ミリ秒窓）による記録漏れの理論的可能性が残る

### トレードオフ

- switch単体での自己修復性を放棄し、明示的な `bswitch release --all`（全プロジェクトスイープ）に回復を委ねる
- 継続スキップはstate.jsonを信頼するため、Backlog UI側で手動除名された場合に「参加しているつもりで実際は未参加」となり得る（`bswitch status` で実状態を確認可能）

## コンプライアンス

- 除名・参加の更新系API操作が成功したら、必ず直後に `save` コールバックを呼んでstate.jsonへ永続化する（switch / release / enforce_expired 共通の実装規約）
- config定義外のプロジェクトにはAPI操作をしない（設計上の制約。記録の削除のみ行い、警告を表示する）
- 更新系API呼び出しを新たに追加する場合は、レートリミット枠（更新系）への影響を必ず見積もる
- 継続スキップ・こまめな保存の挙動は `tests/unit/test_switcher.py` の該当テスト群（継続スキップ5件・途中保存3件）で担保する。仕様変更時はテストも同時に更新する
