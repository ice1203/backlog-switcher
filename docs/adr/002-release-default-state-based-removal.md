# ADR-002: releaseの既定動作をstate.json基準の除名に変更し、全プロジェクトスイープを--allへ分離

## ステータス
承諾済み

## 日付
2026-07-18

## コンテキスト

`bswitch release` は当初「config定義済みの**全**プロジェクトから両仮想ユーザーを除名する」コマンドとして実装されていた（ADR-001の回復経路）。この全走査は プロジェクト数 × 2ユーザー × 2操作（ユーザー除名・管理者除名）= **n*4回の更新系APIコール** を毎回消費する。

一方、releaseの主用途は「作業終了時に参加中の権限を外す」ことであり、通常時に実際へ参加しているのはstate.jsonのgrantsに記録された1〜2プロジェクトのみである。日常操作のたびにconfig全体を走査するのは、ADR-001で最も希少なリソースと位置づけた更新系レートリミット枠（150回/時）の浪費になる。そこで「state.jsonに記録された参加中プロジェクトのみから除名する」動作の導入が必要になった。

なお `enforce` については同様の懸念が提起されたが、実装確認の結果、`enforce_expired()` は元からstate.jsonのgrantsのみを対象とし（期限切れ0件ならAPIコール0回）、config全走査はしていないため変更不要と確認した。

### 検討した選択肢

1. **releaseにフラグ追加（既定は全走査を維持）**
   - 概要: 既定は従来どおりconfig全プロジェクト走査。`--granted` 等のフラグでstate.json基準の軽量版を選ぶ
   - メリット: 「releaseすれば確実に全部外れる」という既存セマンティクス・回復経路の既定を維持でき、後方互換
   - デメリット: 日常の作業終了時release（最頻の利用形態）が引き続きn*4回の更新系APIコールを消費する。軽量版はフラグを知らないと使われない

2. **releaseの既定を反転（採用）**
   - 概要: `bswitch release` をstate.json基準（grants記載プロジェクトのみ除名）に変更し、config全プロジェクト走査は `--all` フラグへ移す
   - メリット: 最頻の利用形態が最小APIコール（参加中プロジェクト数×4回、付与記録0件なら0回）になる。回復経路は `--all` として明示的に残る
   - デメリット: 「releaseは全部外す」という従来の動作が変わり、README・ADR-001の記述更新が必要。state.json消失時は既定releaseでは残留参加を回収できない

3. **サブコマンド新設**
   - 概要: `bswitch revoke` 等の別サブコマンドとしてstate.json基準の除名を追加する
   - メリット: releaseの動作を一切変えずに済む
   - デメリット: 操作の本質（除名 + grants削除 + unset行出力）は同一で走査範囲だけが違うため、コマンドを分ける必然性がない。help・README・シェル統合ドキュメントの表面積だけが増える

## 決定

私たちは **案2（releaseの既定をstate.json基準に反転し、全プロジェクトスイープを `--all` へ分離）** を使用します。

### 決定理由

- releaseの最頻の利用形態は作業終了時の後片付けであり、既定動作をそこへ最適化するのがレートリミット優先の設計方針（ADR-001）と一貫する
- 全プロジェクトスイープの本来の役割は「state.json消失等の異常時の回復」であり、日常の既定動作ではなく明示的なオプション（`--all`）として提供するのが役割に合う
- モードの違いは走査範囲のみであるため、サブコマンドではなくフラグで表現するのが自然（案3の却下理由）

### 付随する決定

- **既定モードの安全ガード**: state.json基準の除名でも、除名は `_exclusive_remove`（両仮想ユーザーのプロジェクトユーザー・プロジェクト管理者の4操作）を再利用し、grantsに記録されたユーザーだけでなく両ユーザーを掃除する（安全側）
- **config定義外プロジェクトの扱い**: grants記載のプロジェクトが現在のconfigに存在しない場合、API操作をスキップし警告表示のうえ記録のみ削除する（switchの排他除名と同一のガード）
- **ゼロコール最適化**: 付与記録が0件の既定releaseは、ユーザーID解決も含めAPIを一切呼ばずunset行のみ出力する
- **e2eの後片付け**: E2Eテストの最終ステップはgrants記録に依存しない `release --all` を使用する

## 結果

### ポジティブ

- 作業終了時のreleaseの更新系APIコール: 旧n*4回 → 参加中プロジェクト数×4回（通常4回）、付与記録0件なら**0回**
- レートリミット消費がプロファイル数の増加に比例しなくなる
- 回復経路（全プロジェクトスイープ）は `--all` として明示的に維持され、ADR-001の残余リスク対策は失われない

### ネガティブ

- 「releaseすれば確実に全部外れる」という従来の習慣の意味が変わる（既存利用者は `--all` の存在を知る必要がある）
- state.json消失・破損時、既定のreleaseは残留参加を検知・回収できない（`--all` の明示実行が必要）
- Backlog UI側で手動追加された参加（grants記録なし）は既定のreleaseでは回収されない

### トレードオフ

- 常時全掃除による自己修復性よりも、日常のレートリミット節約を既定に据える。残留参加が疑われる場合の回収は利用者の明示判断（`release --all`）に委ねる

## コンプライアンス

- 既定のreleaseはstate.jsonのgrants記載プロジェクトのみを対象とし、config定義外プロジェクトにはAPI操作をしない（記録削除+警告のみ）
- 全プロジェクトスイープを行う経路は `release --all` のみとし、ドキュメント（README・ADR-001）では回復経路として `--all` を案内する
- ADR-001の「こまめな保存」規約（1プロジェクト分の除名完了ごとに `save` コールバックで永続化）を両モードで維持する
- 挙動は `tests/unit/test_switcher.py`（既定モード3件・`--all` 1件）および `tests/unit/test_cli.py`（既定2件・`--all` 1件）で担保する。仕様変更時はテストも同時に更新する
