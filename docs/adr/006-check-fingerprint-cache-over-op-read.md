# ADR-006: `bswitch check` のAPIキー照合を `op read` からフィンガープリントキャッシュに変更する

## ステータス
承諾済み（ADR-005 の「check 実装における `op read` 利用」を部分的に置き換え）

## 日付
2026-07-21

## コンテキスト

ADR-005 で設計した `bswitch check` は、現在の `BACKLOG_API_KEY` と期待値を照合するために
`resolve_api_key()` 経由で `op read` を呼び出していた。
ADR-005 設計時は「`op read` はステータスライン更新のたびに呼ばれる可能性があり、
1Password ロック中は整合チェックが失敗する」という懸念を認識していたが、
実運用で想定より深刻な問題が発覚した。

**実際に発生した問題**: `op read` は毎回 1Password の認証ダイアログを起動する。
ステータスライン（Claude Code / tmux / starship）が数秒ごとに `bswitch check` を呼ぶ構成では、
画面に認証ダイアログが常時ポップアップし続けてステータスライン機能が実用不能になる。
1Password デスクトップアプリとの IPC 連携でも認証要求は回避できなかった。

本質的な問題は「check のたびに `op read` を呼ぶ」設計にある。
`op read` を呼ぶ唯一の目的は「現在の `BACKLOG_API_KEY` が期待値と一致するか」の比較であり、
比較のためにフルキーを取得する必要はない。

### 検討した選択肢

1. **`op read` をそのまま維持（変更なし）**
   - 概要: 設計変更を行わない
   - メリット: 実装コストゼロ
   - デメリット: ステータスライン更新のたびに認証ダイアログが表示され実用不能。根本解決にならない

2. **MISMATCH 検出を廃止する**
   - 概要: `bswitch check` を「グラントの有無」と「`BACKLOG_API_KEY` の設定有無」のみを返す設計に変更し、`op read` 呼び出しを排除する
   - メリット: `op read` を完全に排除でき、実装もシンプル
   - デメリット: 「`BACKLOG_API_KEY` が期待値と一致するか」の検出がなくなり、キーの乖離を検知できない。本ツールの整合チェックという主要要件を損なう

3. **フルキーの SHA-256 ハッシュを state に保存する**
   - 概要: `bswitch switch` 実行時にフルキーの `sha256(key)` を state.json に保存し、`bswitch check` はハッシュ同士で比較する
   - メリット: `op read` 不要。高エントロピーキーに対してハッシュは実質不可逆
   - デメリット: ハッシュが万一逆算された場合（低エントロピーキーや将来の計算能力向上）にフルキーが復元されるリスクが理論上存在する

4. **先頭・末尾5文字の SHA-256 ハッシュを state に保存する（`key_fingerprint`）**
   - 概要: `bswitch switch` 実行時に `sha256(key[:5] + key[-5:])` を計算し `key_fingerprint` として state.json に保存。`bswitch check` はこのハッシュと `sha256(BACKLOG_API_KEY[:5] + BACKLOG_API_KEY[-5:])` を比較する
   - メリット: `op read` 不要。万一ハッシュが逆算されてもフルキーへの復元が不可能（逆算で得られるのは先頭・末尾5文字のみ）。フルキーより多層防御になる
   - デメリット: 中間部のみが異なるキーを誤って OK と判定する可能性がある（ただし実運用で自然発生する確率はほぼゼロ）

## 決定

私たちは **先頭・末尾5文字の SHA-256 ハッシュ（`key_fingerprint`）を state に保存し、`bswitch check` は `op read` を呼ばずにフィンガープリント比較で整合チェックを行う** 設計を採用します。

### 決定理由

- `op read` の呼び出しは認証ダイアログを発生させるため、ステータスライン用途に根本的に不適合であり、選択肢 1 は排除
- MISMATCH 検出は「`state.json` の記録と現在の環境が乖離していないか」を知る主要要件であり廃止できない（選択肢 2 は排除）
- フルキーハッシュ（選択肢 3）と先頭・末尾5文字ハッシュ（選択肢 4）は実用上ほぼ同等のセキュリティだが、選択肢 4 は「万一ハッシュが逆算された場合でもフルキーに戻せない」という多層防御の性質を持つ
- `bswitch switch` の実行時点でキーはすでにメモリ上にあるため、フィンガープリント計算に追加の `op read` は不要

### セキュリティ分析（選択肢 4）

- ハッシュ対象は10文字（英数字 36 種）: 探索空間 36^10 ≈ 3.6 兆通り → ブルートフォースは現実的だが、得られるのは先頭・末尾5文字のみ
- フルキー（20 文字以上のランダム文字列）への逆算は不可能
- state.json を読める攻撃者はすでにシェル環境変数（`BACKLOG_API_KEY` 本体）も読める状況であり、フィンガープリントから得られる情報の増分は実質ゼロ
- bswitch が扱うキー同士（writer / reader、複数プロファイル）で先頭・末尾5文字が偶然一致する確率はほぼゼロ

### `UNKNOWN` ステータスの導入

`bswitch switch` を以前のバージョンで実行したグラントには `key_fingerprint` が存在しない。
このケースを「マイグレーション目的の複雑なロジック」で扱うのではなく、
`UNKNOWN` という新しいステータスを導入してシンプルに対処する（`bswitch switch` の再実行を促す）。

## 結果

### ポジティブ
- ステータスライン更新時に 1Password 認証ダイアログが表示されなくなる
- `bswitch check` の実行に 1Password の起動・認証が一切不要になる
- `op read` 呼び出しを含む `resolve_api_key()` の依存が `_handle_check()` から排除され、`bswitch check` が真に軽量なコマンドになる
- フィンガープリントキャッシュはフルキーより多層防御になる

### ネガティブ
- キーの中間部のみが異なる場合（手動で一部書き換えるなど）の MISMATCH 検出ができなくなる（実用上は無視できる）
- 旧バージョンで切り替えたグラントは `UNKNOWN` として表示され、`bswitch switch` の再実行が必要

### トレードオフ
- フルキー比較による完全な MISMATCH 検出を捨てる代わりに、ステータスライン実用性と多層防御セキュリティを得る
- `UNKNOWN` によるマイグレーション誘導はシンプルだが、再 switch の手間をユーザーに求める

## コンプライアンス
- `_handle_check()` 内で `resolve_api_key()` または `op read` を呼び出してはならない
- `bswitch switch` の完了時（`resolve_api_key()` でキーを取得した直後）に `compute_fingerprint(api_key)` を計算し、全 new_grants の `key_fingerprint` フィールドに設定すること
- `key_fingerprint` の計算は `sha256(key[:5] + key[-5:]).hexdigest()`（64 文字 hex）とすること
- `key_fingerprint` が `None` のグラントは `UNKNOWN` ステータスを返すこと
- state.json の `key_fingerprint` フィールドにはハッシュ値のみを保存し、APIキーの値・部分文字列は絶対に含めないこと
